1 Singleton Pattern 单例模式

GoF Definition

确保一个类只有一个实例，并提供对它的全局访问点

Concept

一个类不能有多个实例。一旦创建，下一次，您将只使用现有的实例。

这种方法可以帮助您限制集中式系统中不必要的对象创建

这种方法还有助于简化维护

现实生活中的一个例子

让我们假设你是一个运动队的成员，你的队伍正在参加一场锦标赛。你的球队需要在整个比赛中与多个对手比赛

在每一场比赛之前，根据游戏规则，双方的队长必须掷硬币。如果你的球队没有队长，你需要选出一个队长

在每场比赛和每次抛硬币之前，如果你已经为球队提名了一个队长，你就不能重复选举队长的过程

基本上，从每个团队成员的角度来看，团队应该只有一个队长

Computer Example

在某些特定的软件系统中，您可能倾向于只使用一个文件系统来集中管理资源。此外，该模式还可以实现缓存机制。

15 Strategy (Policy) Pattern策略(政策)模式

概念

定义一系列算法，封装每一个算法，并使它们可互换。策略让算法独立于使用它的客户机而变化。

策略模式建议您在单独的类中实现这些算法。当您在一个单独的类中封装一个算法时，您称它为策略。

使用策略对象的对象通常称为上下文对象。这些“算法”在某些应用程序中也称为行为。

例子

通常在足球比赛结束时，如果a队以1比0领先B队，他们就会变成守势，而不是进攻，以保持领先。另一方面，B队则全力进攻以扳平比分。

What are the key advantages of using a strategy design pattern?

This pattern makes your classes independent from algorithms. Here, a class delegates the algorithms to the strategy object (that encapsulates the algorithm) dynamically at runtime. So, you can simply say that the choice of the algorithm is not bound at compile time

这个模式使你的类独立于算法。在这里，类在运行时动态地将算法委托给策略对象(封装算法)。因此，您可以简单地说，算法的选择在编译时不受约束

Easier maintenance of your codebase.

It is easily extendable.

What are key challenges associated with a strategy design pattern?

The addition of context classes causes more objects in our application.

上下文类的添加导致应用程序中出现了更多的对象。

Users of the application must be aware of different strategies; otherwise, the output may surprise them. So, there exists a tight coupling between the client code and the implementation of different strategies.

应用程序的用户必须知道不同的策略;否则，输出结果可能会出乎他们的意料。因此，客户端代码与不同策略的实现之间存在紧密耦合。

When you introduce a new behavior/algorithm, you may need to change the client code also.

当您引入一个新的行为/算法时，您可能也需要更改客户机代码。

16Template Method Pattern模板方法模式

概念

在操作中定义算法的框架，将一些步骤推迟到子类中。模板方法允许子类在不改变算法结构的情况下重新定义算法的某些步骤。

在模板方法中，定义算法的最小或基本结构。

然后您将一些责任推给子类。其主要目的是您可以重新定义算法的某些步骤，但这些更改不应影响算法的基本流程。

因此，当您实现多步骤算法，并且希望通过子类进行定制时，这种设计模式非常有用。

例子

假设你在一家餐馆订了一份披萨。对于厨师来说，制作披萨的基本方法是一样的;他还根据顾客的选择添加了一些配料。例如，你可以选择素食披萨或非素食披萨。你也可以选择像培根，洋葱，额外的奶酪，蘑菇，等等。厨师根据你的喜好准备最后的成品。

图示

AbstractClass 实现了一个模板方法，定义了算法的骨架，而逻辑的组成步骤在相应的抽象操作中，推迟到了子类实现。顶级逻辑也有可能调用一些具体方法。

ConcreteClass,实现父类所定义的一个或多个抽象方法，每一个AbstractClass 都可以有任意的ConcreteClass与之对应，而每一个ConcreteClass都可以给出这些抽象方法的不同实现，从而使顶级逻辑的实现各不相同。

What are the key advantages of using a template design pattern?

You can control the flow of the algorithms. Clients cannot change them. (Notice that completeCourse() is marked with final keyword in the abstract class BasicEngineering)

你可以控制算法的流程。客户端不能更改它们。(注意，completeccourse()在抽象类BasicEngineering中被标记为final关键字)

Common operations are placed in a centralized location, for example, in an abstract class. The subclasses can redefine only the varying parts, so that, you can avoid redundant codes.

公共操作被放在一个集中的位置，例如，在一个抽象类中。子类只能重新定义变化的部分，这样就可以避免冗余代码。

What are key challenges associated with a template design pattern?

Client code cannot direct the sequence of steps (If you need that approach, you may follow the Builder pattern).

客户端代码不能指导步骤的顺序(如果需要这种方法，可以遵循Builder模式)。

A subclass can override a method defined in the parent class (i.e. hiding the original definition in parent class) which can go against Liskov Substitution Principle that basically says: If S is a subtype of T, then objects of type T can be replaced with objects of type

子类可以覆盖父类中定义的方法(即在父类中隐藏原始定义)，这违背了Liskov替换原则，即:如果S是T的子类型，那么T类型的对象可以被类型对象替换

More subclass means more scattered codes and difficult maintenance

更多的子类意味着更分散的代码和更难维护

17 Command Pattern命令模式

概念

将请求封装为对象，从而使您可以用不同的请求、队列或日志请求对客户端进行参数化，并支持可撤销操作。

命令对象可以以特定于接收方类的方式调用接收方的方法。然后，接收方开始处理该作业。命令对象被单独传递给调用程序对象以调用该命令。客户端对象保存调用程序对象和命令对象。客户机只做出决策——执行哪个命令——然后将命令传递给调用者对象(用于执行该命令)。

例子

Why do you need the invoker?

大多数时候，程序员尝试在面向对象编程中封装数据和相应的方法。但是如果仔细观察，就会发现在这个模式中，您试图封装命令对象。换句话说，您正在从不同的角度实现封装。当您处理一组复杂的命令时，这种方法很有意义。现在我们再来复习一下条款。您创建命令对象，将它们发送给接收者并调用一些方法。但是您通过调用程序执行这些命令，调用命令对象的方法(例如，executeccommand)。但是对于简单的情况，这个调用程序类不是强制性的;例如，考虑这样一种情况，命令对象只有一个要执行的方法，而您试图不用调用程序来调用该方法。但是，当您希望在日志文件(或队列)中跟踪多个命令时，调用程序可能会发挥重要作用

What are the key advantages associated with command patterns?

Requests for creation and the ultimate execution are decoupled. Clients may not know how an invoker is performing the operations.

创建请求和最终执行请求是解耦的。客户机可能不知道调用者是如何执行操作的。

You can create macros (sequence of commands).

您可以创建宏(命令序列)。

New commands can be added without affecting the existing system.

以在不影响现有系统的情况下添加新的命令。

Most importantly, you can support the undo/redo operations.

最重要的是，你可以支持撤消/重做操作。

What are the challenges associated with command patterns?

To support more commands, you need to create more classes. So,maintenance can be difficult as time goes on.

要支持更多的命令，您需要创建更多的类。因此，随着时间的推移，维护会变得困难

How to handle errors or make a decision about what to do with return values when an erroneous situation occurs becomes tricky. A client may want to know about those. But here you decouple the command with client codes, so these situations are difficult to handle. The challenge becomes significant in a multithreaded environment where the invoker is also running in a different thread.

当出现错误情况时，如何处理错误或决定如何处理返回值变得非常棘手。客户可能想知道这些。但是这里您将命令与客户机代码解耦，因此这些情况很难处理。在多线程环境中，调用程序也运行在不同的线程中，这个挑战就变得很重要了。

18Iterator Pattern迭代器模式

概念

提供一种顺序访问聚合对象的元素而不公开其底层表示的方法。

例子

假设有两个公司:公司A和公司b。公司A在一个链表数据结构中存储其员工记录(即姓名、地址、工资细节等)。公司B将其员工数据存储在数组数据结构中。一天，这两家公司决定合并，组成一个大公司。迭代器模式在这种情况下很方便，因为开发人员不想从头开始编写代码。他们可以创建一个公共接口，以便以统一的方式访问这两家公司的数据和调用方法。

考虑另一个例子。假设你的公司根据员工的表现决定提拔他们。所以，所有的经理聚在一起，为晋升设定一个共同的标准。然后，他们对每个员工过去的记录进行迭代，以标记出可能的晋升人选。

最后，当您将歌曲存储在您喜欢的音频设备(例如MP3播放器或移动设备)中时，您可以通过各种按钮或滑动移动来迭代它们。其基本思想是为您提供某种机制来平滑地遍历列表。

迭代器模式的用途是什么?

You can traverse an object structure without knowing its internal details. As a result, if you have a collection of different subcollections (e.g., your container is mixed up with arrays, lists, or linked lists, etc.), you can still traverse the overall collection and deal with the elements in a universal way without knowing the internal details or differences among them.

你可以遍历一个对象结构而不知道它的内部细节。因此，如果你有一个不同子集合的集合(例如，你的容器混合了数组、列表或链表等)，你仍然可以遍历整个集合并以通用的方式处理元素，而不需要知道内部细节或它们之间的差异。

You can traverse a collection in different ways. You can also provide an implementation that supports multiple traversals simultaneously.

您可以以不同的方式遍历一个集合。您还可以提供同时支持多个遍历的实现

与此模式相关的主要挑战是什么?

Ideally, during a traversal/iteration process, you should not perform any accidental modification to the core architecture.

理想情况下，在遍历/迭代过程中，您不应该对核心架构执行任何偶然的修改。

1.使用原型设计模式的优点是什么?

  • 当创建一个实例的类是复杂的(或无趣)的过程中它是有用的。相反，你可以专注于其他重要的活动。

  • 您可以在运行时包含或丢弃产品。

  • 您可以以较低的成本创建新实例。

2.使用原型设计模式有哪些挑战?

  • 每个子类都需要实现克隆或复制机制。

  • 有时从现有实例创建副本并不简单。例如，如果考虑的对象不支持复制/克隆，或者存在循环引用，那么实现克隆机制就很有挑战性。例如，在Java中，带有clone()方法的类需要实现Cloneable标记接口;否则，它将抛出一个

3.你能详细说明浅拷贝和深拷贝的区别吗?

  • 浅拷贝创建一个新对象，然后将不同的字段值从原始对象复制到新对象。因此，它也被称为逐字段复制。如果原始对象包含任何对其他对象的引用作为字段，那么这些对象的引用将被复制到新对象中(即，您不创建这些对象的副本)。

    让我们试着用一个简单的图表来理解它的机理。假设我们有一个对象X1，它有一个指向另一个对象Y1的引用。进一步假设对象Y1有对象Z1的引用。

    现在，对于X1的浅拷贝，创建了一个新对象X2;它还有一个对Y1的引用。

    浅层复制后的引用

    您已经看到了在我们的实现中使用clone()方法。它执行浅拷贝。

  • 对于X1的深度拷贝，创建了一个新对象X3。X3有一个对新对象Y3的引用，它实际上是Y1的一个副本。同样，Y3也有一个对另一个新对象Z3的引用，它是Z1的副本。

    在深度复制中，新对象与原对象完全分离。对一个对象所做的任何更改都不应反映到另一个对象上。要在Java中创建深度复制，可能需要重写clone()方法，然后继续。此外，深度复制的成本很高，因为您需要创建额外的对象.深度复制的完整实现在本书的Memento Pattern(第19章)的“Q&A Session”中给出

浅拷贝更快，也更便宜。如果你的目标对象只有原始字段总是更好的。

22 Chain-of-Responsibility Pattern职责链模式

概念

通过给多个对象处理请求的机会，避免将请求的发送方与接收方耦合。将接收对象连接起来，并沿着链传递请求，直到有一个对象处理它。

如果一个对象不能完全处理请求，它将把请求传递给链中的下一个对象。同样的过程可能会继续下去，直到到达链的末端。这种请求处理机制使您能够灵活地在链的末端添加一个新的处理对象(处理程序)。

What are the advantages of using a chain-of-responsibility design pattern?

您可以有多个对象来处理一个请求。(请注意，如果一个处理程序不能处理整个请求，它可能会将责任转发给链中的下一个处理程序)。

You can have more than one object to handle a request. (Notice that if a handler cannot handle the whole request, it may forward the responsibility to the next handler in the chain).

链的节点可以动态添加或删除。

The nodes of the chain can be added or removed dynamically.

另外，你可以打乱顺序。例如，如果你注意到大部分的问题是与电子邮件处理，那么你可以放EmailErrorHandler作为链中的第一个处理程序，以节省应用程序的平均处理时间。

Also, you can shuffle the order. For example, if you notice that the majority of issues are with email processing, then you may place EmailErrorHandler as the first handler in the chain to save the average processing time of the application.

一个处理程序不需要知道链中的下一个处理程序将如何处理请求。

A handler does not need to know how the next handler in the chain will handle the request. It focuses only on its own handling mechanism.

在此模式中，您正在促进松散耦合，因为它将发送方(请求的)与接收方解耦。

In this pattern, you are promoting loose coupling because it decouples the senders (of requests) from the receivers.

What are the challenges associated with using the chain-of-responsibility design pattern?

不能保证请求会被处理(完全或部分)，因为你可能会到达链的末端;但也有可能您没有找到任何显式的接收方来处理请求。

There is no guarantee that the request will be handled (fully or partially) because you may reach the end of the chain; but it is possible that you have not found any explicit receiver to handle the request.

在这种设计下，调试可能会变得棘手。

Debugging may become tricky with this kind of design.

similarities between the observer pattern and the chain-of-responsibility pattern.

In an observer pattern, all registered users get notifications in parallel; but in a chain-of-responsibility pattern, objects in the chain are notified, one by one, in a sequential manner. This process continues until an object handles the notification fully (or you reach the end of the chain). I show the comparisons in diagrams in the “Q&A Session” in Chapter 14.

在观察者模式中，所有注册用户都会并行地获得通知；但是在责任链模式中，链中的对象会以顺序的方式一个接一个地被通知。此过程一直持续，直到对象完全处理通知（或到达链的末端）。我在第14章的“问答环节”中展示了图表中的比较。
